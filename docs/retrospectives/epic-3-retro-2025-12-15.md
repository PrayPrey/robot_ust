### 🚧 Challenges and Growth Areas

**Blockers Encountered:**

1. **E2E Test Pass Rate (4 hours debugging)**
   - **Problem:** 4/10 E2E tests failing after Story 3.6 fixes
   - **Root cause:** Architectural issues beyond bug-fix scope
   - **Resolution:** Documented as technical debt for future epic
   - **Learning:** E2E scenarios should inform architecture design from start

2. **Mock Sensor Fixture Mismatch (30 min fix)**
   - **Problem:** Lidar samples 360 vs actual 512 required
   - **Root cause:** Mock data created without verifying actual system spec
   - **Resolution:** Updated fixtures at `test_epic3_e2e.py:157, 171`
   - **Learning:** Mock data must always match real system specifications

3. **Windows Console Encoding (15 min investigation)**
   - **Problem:** Korean commands displayed incorrectly (CP949 console issue)
   - **Root cause:** Windows console encoding, not data storage
   - **Resolution:** Confirmed UTF-8 data stored correctly; display-only issue
   - **Learning:** Distinguish between storage vs display encoding issues

4. **format_rag_context() Parameter Mismatch (1 hour fix)**
   - **Problem:** Replanning failed due to incorrect parameter extraction
   - **Root cause:** Search results structure changed without updating caller
   - **Resolution:** Fixed extraction at `src/agents/planner_agent.py:519-523`
   - **Learning:** Integration points need explicit interface contracts

**Process Gaps:**

5. **E2E Tests as Afterthought (John)**
   - Integration testing came in Story 3.5 (after main implementation)
   - Architectural issues discovered too late to fix in scope
   - **Learning:** Define E2E scenarios before implementation starts

6. **WebSocket Testing Complexity (Murat)**
   - Real-time WebSocket testing proved challenging (68% pass rate in Story 3.2)
   - Async test patterns and connection lifecycle hard to mock
   - **Learning:** Dedicated time for real-time system test infrastructure

### 💡 Key Insights

**Pattern to Repeat:**
- **Rule-based environment classification** - Simple, interpretable, effective
- **RAG metadata extension pattern** - Add fields to existing schema vs restructuring
- **3-level priority system** - CRITICAL/MODERATE/NORMAL covers most use cases
- **Early scope decisions** - Defer non-essential stories (3.4) to maintain focus

**Pattern to Avoid:**
- **E2E scenarios as afterthought** - Define end-to-end flows before coding
- **Mock data without spec verification** - Always match real system specifications
- **Architectural fixes as bug tickets** - Structural issues need dedicated refactoring
- **Integration testing last** - Move integration stories earlier in epic

**Skills Gained:**
- **FastAPI WebSocket patterns** - ConnectionManager, broadcast, lifecycle
- **Ollama local LLM integration** - API, model selection, performance tuning
- **Rule-based classification systems** - Threshold-based decision trees
- **E2E test scenario design** - Comprehensive workflow coverage strategies
